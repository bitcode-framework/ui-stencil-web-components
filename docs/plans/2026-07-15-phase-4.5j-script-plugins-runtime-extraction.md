# Phase 4.5j: Script Plugin Interface + Runtime Extraction

**Date**: 15 July 2026
**Status**: Draft
**Depends On**: Phase 4.5i (Advanced Features), Phase 2 (Node.js), Phase 3 (Python), Phase 4 (goja), Phase 5 (yaegi)
**Effort**: 3-4 weeks
**Scope**: `packages/go-json/runtime/`, `packages/go-json/lang/`, `packages/go-json-runtimes/` (NEW), `engine/internal/runtime/embedded/`

---

## 1. Overview

This phase accomplishes two major goals:

1. **Define `ScriptRuntime` interface in go-json core** — enables go-json programs to call external scripts (Python, Node.js, Go, etc.) via a first-class `script:` import type.
2. **Extract runtime engines into `packages/go-json-runtimes/`** — decouple all runtime implementations (goja, quickjs, yaegi, Node.js, Python) from BitCode engine into a standalone, independently-importable package.

After this phase:
- go-json core remains zero-dependency (defines interface only)
- `go-json-runtimes` is a separate `go.mod` that implements the interface
- BitCode engine imports `go-json-runtimes` instead of maintaining its own runtime code
- go-json programs can call external scripts via `script:` imports

---

## 2. Goals & Non-Goals

### Goals

- Define `ScriptRuntime` interface in go-json core (`packages/go-json/runtime/`)
- Add `script:` import type to go-json parser and VM
- Create `packages/go-json-runtimes/` with separate `go.mod`
- Extract goja, quickjs, yaegi, Node.js, Python runtimes from BitCode
- Decouple VM interface from `*bridge.Context` → accept `map[string]any`
- Implement configurable pool manager in go-json-runtimes
- Maintain backward compatibility (BitCode external API unchanged)
- All existing tests continue to pass

### Non-Goals

- WASM runtime (Phase 4.5k)
- Native plugin (.so/.dll) support (Phase 4.5k)
- New runtime engines (Deno, Bun as separate runtime, etc.)
- Changing BitCode's process JSON format
- Modifying how BitCode selects runtimes for process steps

## 4. Interface Definitions

### 4.1 ScriptRuntime Interface (go-json core)

```go
// packages/go-json/runtime/script_runtime.go

package runtime

import "context"

// ScriptRuntime defines the interface for external script execution engines.
// Implementations live in go-json-runtimes package.
// go-json core defines this interface but never imports implementations.
type ScriptRuntime interface {
    // Name returns the runtime identifier (e.g., "python", "node", "goja").
    Name() string

    // Extensions returns file extensions this runtime can handle (e.g., [".py", ".pyw"]).
    Extensions() []string

    // CanHandle reports whether this runtime can execute scripts with the given extension.
    CanHandle(extension string) bool

    // Execute runs a script file with the given parameters and bridge.
    // - ctx: execution context (for cancellation/timeout)
    // - script: path to the script file (absolute or relative to working dir)
    // - function: function name to call within the script (empty = execute entire script)
    // - params: input parameters passed to the script as JSON-compatible map
    // - bridge: host functions exposed to the script (map[string]any, not typed)
    // Returns the script's return value (JSON-compatible) or error.
    Execute(ctx context.Context, script string, function string, params map[string]any, bridge map[string]any) (any, error)

    // Validate checks if the runtime is available (e.g., Python installed, correct version).
    // Returns nil if ready, descriptive error if not.
    Validate() error

    // Close releases all resources held by this runtime (process pools, VMs, etc.).
    Close() error
}

// ScriptRuntimeRegistry manages multiple ScriptRuntime implementations.
type ScriptRuntimeRegistry struct {
    runtimes []ScriptRuntime
}

// NewScriptRuntimeRegistry creates an empty registry.
func NewScriptRuntimeRegistry() *ScriptRuntimeRegistry {
    return &ScriptRuntimeRegistry{}
}

// Register adds a runtime to the registry.
func (r *ScriptRuntimeRegistry) Register(rt ScriptRuntime) {
    r.runtimes = append(r.runtimes, rt)
}

// Resolve finds a runtime that can handle the given file extension.
// Returns nil if no runtime can handle it.
func (r *ScriptRuntimeRegistry) Resolve(extension string) ScriptRuntime {
    for _, rt := range r.runtimes {
        if rt.CanHandle(extension) {
            return rt
        }
    }
    return nil
}

// ResolveByName finds a runtime by name.
func (r *ScriptRuntimeRegistry) ResolveByName(name string) ScriptRuntime {
    for _, rt := range r.runtimes {
        if rt.Name() == name {
            return rt
        }
    }
    return nil
}

// All returns all registered runtimes.
func (r *ScriptRuntimeRegistry) All() []ScriptRuntime {
    return r.runtimes
}

// Close closes all registered runtimes.
func (r *ScriptRuntimeRegistry) Close() error {
    var firstErr error
    for _, rt := range r.runtimes {
        if err := rt.Close(); err != nil && firstErr == nil {
            firstErr = err
        }
    }
    return firstErr
}
```

### 4.3 VM Interface (go-json-runtimes)

```go
// packages/go-json-runtimes/runtime.go

package runtimes

import (
    "context"
    "time"
)

// EmbeddedRuntime represents an embeddable script engine (goja, quickjs, yaegi).
type EmbeddedRuntime interface {
    Name() string
    NewVM(opts VMOptions) (VM, error)
}

// VM represents a single execution instance of an embedded runtime.
type VM interface {
    // InjectBridge exposes host functions to the script.
    // bridge is map[string]any — decoupled from any framework-specific types.
    InjectBridge(bridge map[string]any) error

    // InjectParams sets input parameters for the script.
    InjectParams(params map[string]any) error

    // Execute runs the given code and returns the result.
    Execute(code string, filename string) (any, error)

    // Interrupt signals the VM to stop execution.
    Interrupt(reason string)

    // Close releases VM resources.
    Close()
}

// VMOptions configures a new VM instance.
type VMOptions struct {
    Timeout      time.Duration
    MaxMemoryMB  int
    HardMaxMemMB int
}

// ExternalRuntime represents a child-process script engine (Node.js, Python).
type ExternalRuntime interface {
    Name() string
    Execute(ctx context.Context, script string, function string, params map[string]any, bridge map[string]any) (any, error)
    Validate() error
    Close() error
}
```

### 4.4 Bridge Contract

The bridge is always `map[string]any`. This is the same format that go-json's `ext:` extensions already use. The contract:

```go
// Bridge structure (built by host — BitCode or standalone app):
bridge := map[string]any{
    "model": func(name string) any { ... },
    "db": map[string]any{
        "query":   func(sql string, params ...any) (any, error) { ... },
        "execute": func(sql string, params ...any) (any, error) { ... },
    },
    "http": map[string]any{
        "get":  func(url string, opts ...any) (any, error) { ... },
        "post": func(url string, opts ...any) (any, error) { ... },
        // ...
    },
    "cache": map[string]any{ ... },
    "fs":    map[string]any{ ... },
    "emit":  func(event string, data ...any) (any, error) { ... },
    "call":  func(process string, input ...any) (any, error) { ... },
    "log":   func(level string, msg string, data ...any) (any, error) { ... },
    // ...
}
```

For embedded runtimes (goja, yaegi): bridge functions are directly callable from the script.
For external runtimes (Node.js, Python): bridge function catalog is serialized, and actual calls go through JSON-RPC.

## 5. `script:` Import Type

### 5.1 Syntax

```jsonc
{
  "import": {
    "ml": "script:./plugins/predict.py",
    "utils": "script:./helpers/utils.js",
    "fast": "script:./compute/heavy.go"
  },
  "steps": [
    {"let": "prediction", "expr": "ml.predict(features)"},
    {"let": "formatted", "expr": "utils.format(prediction)"},
    {"let": "result", "expr": "fast.compute(data)"}
  ]
}
```

### 5.2 Resolution Rules

1. Path is relative to the importing program's directory (same as `./` imports)
2. File extension determines which `ScriptRuntime` handles execution
3. If no runtime can handle the extension → compile-time error
4. If runtime is not available (e.g., Python not installed) → runtime error with clear message

### 5.3 Parser Changes

```go
// packages/go-json/lang/parser.go — import type detection

// Existing path types: "relative" (./), "stdlib" (stdlib:), "io" (io:), "ext" (ext:)
// NEW: "script" (script:)

func classifyImportPath(path string) (pathType string, resolvedPath string) {
    switch {
    case strings.HasPrefix(path, "./") || strings.HasPrefix(path, "../"):
        return "relative", path
    case strings.HasPrefix(path, "stdlib:"):
        return "stdlib", strings.TrimPrefix(path, "stdlib:")
    case strings.HasPrefix(path, "io:"):
        return "io", path
    case strings.HasPrefix(path, "ext:"):
        return "ext", path
    case strings.HasPrefix(path, "script:"):
        return "script", strings.TrimPrefix(path, "script:")
    default:
        return "relative", path
    }
}
```

### 5.5 VM Execution

When the VM encounters a call to a script-imported namespace:

```go
// packages/go-json/lang/vm.go — script: call handling

// 1. Expression "ml.predict(features)" is evaluated by expr-lang
// 2. "ml" resolves to a ScriptNamespace proxy in the scope
// 3. ScriptNamespace.predict(features) → calls ScriptRuntime.Execute(...)

// ScriptNamespace is injected into scope during Execute():
type scriptNamespace struct {
    runtime  ScriptRuntime
    script   string   // resolved absolute path
    bridge   map[string]any
}

// When expr-lang calls a method on scriptNamespace, it goes through:
func (ns *scriptNamespace) Call(function string, args ...any) (any, error) {
    ctx := context.Background() // or from VM context
    params := map[string]any{"args": args}
    return ns.runtime.Execute(ctx, ns.script, function, params, ns.bridge)
}
```

### 5.7 Script Call Design (Two Tiers)

Script calls work at two levels. Tier 1 (step-level) MUST work in Phase 4.5j. Tier 2 (expression-level) SHOULD work if expr-lang supports it, otherwise deferred.

#### Tier 1: Step-Level Calls (MUST work)

Uses the existing `call` step mechanism. The VM intercepts dotted names that match script imports:

```jsonc
{"let": "result", "call": "ml.predict", "with": {"features": "data"}}
```

VM resolution:
1. Split `"ml.predict"` on first `.` → namespace=`"ml"`, function=`"predict"`
2. Check if `"ml"` is a script import in `AST.RequestedModules`
3. If yes → `ScriptRuntime.Execute(ctx, scriptPath, "predict", params, bridge)`
4. If no → fall through to existing function resolution

This requires NO changes to expr-lang. Only a small VM change in the `call` step handler.

#### Tier 2: Expression-Level Calls (SHOULD work)

```jsonc
{"let": "result", "expr": "ml.call('predict', features)"}
```

The script namespace is injected into scope as a `map[string]any` with a `call` function:

```go
func buildScriptProxy(rt ScriptRuntime, scriptPath string, bridge map[string]any, ctx context.Context) map[string]any {
    return map[string]any{
        "call": func(function string, args ...any) (any, error) {
            params := map[string]any{"args": args}
            return rt.Execute(ctx, scriptPath, function, params, bridge)
        },
    }
}
```

Usage: `ml.call("predict", features)` — works immediately with expr-lang because `ml` is a `map[string]any` and `call` is a Go function value.

#### Future: Dot-Notation Expression Calls

```jsonc
{"let": "result", "expr": "ml.predict(features)"}
```

This requires either:
- expr-lang support for dynamic property access on custom types, OR
- VM-level interception before expr-lang evaluation

This is NOT in scope for Phase 4.5j. If expr-lang supports it via reflection on custom types, the implementing agent may attempt it. Otherwise, `ml.call("predict", ...)` is the expression-level API.

### 5.8 Script Function Discovery

Scripts declare exported functions via language convention:

**Python**: All top-level functions not starting with `_` are exported.
**Node.js**: `module.exports` object keys are exported.
**Go**: Exported (capitalized) functions are exported.

Discovery is **lazy** — the runtime does not scan scripts at import time. Function names are resolved at call time. If a function doesn't exist, the runtime returns a clear error.

### 5.9 Scope Injection Summary

```go

        return nil, fmt.Errorf("no script runtime for extension '%s' (import '%s')", ext, alias)

        return nil, fmt.Errorf("script runtime '%s' not available: %w", rt.Name(), err)
```

## 7. Pool Manager Design

### 7.1 Pool Interface

```go
// packages/go-json-runtimes/pool/pool.go

package pool

import (
    "context"
    "sync"
    "time"
)

// PoolConfig defines configuration for a process pool.
// Maps to bitcode.yaml runtime.worker / runtime.background sections.
type PoolConfig struct {
    Size            int           // pool_size — number of processes in pool
    MaxExecutions   int           // max_executions — recycle process after N executions (0 = never)
    MaxMemoryMB     int           // max_memory_mb — memory limit per process (0 = unlimited). Overridable by module/process.
    HardMaxMemoryMB int           // hard_max_memory_mb — absolute memory ceiling (0 = no ceiling). NOT overridable.
    MaxIdleTime     time.Duration // kill idle process after this duration (0 = never)
    CrashRecovery   bool          // auto-restart crashed processes
    MaxBackoff      time.Duration // max backoff between restart attempts
}

// DualPoolConfig configures worker + background pools.
type DualPoolConfig struct {
    Worker     PoolConfig
    Background PoolConfig
}

// TimeoutConfig holds timeout settings that are enforced by the CALLER (BitCode executor
// or go-json CLI), NOT by the pool manager. go-json-runtimes receives context.Context
// with deadline already set — it respects ctx.Done() but does not set timeouts itself.
//
// These values come from bitcode.yaml and follow the resolution hierarchy:
// step → process → module → project → hardcoded default.
// go-json-runtimes only sees the final resolved value via context.Context.
//
// BitCode populates these from bitcode.yaml for its own executor to use.
// Standalone go-json CLI uses its own defaults (30s step, 5m process).
type TimeoutConfig struct {
    DefaultStepTimeout time.Duration // default timeout per script step (bitcode.yaml: default_step_timeout)
    MaxStepTimeout     time.Duration // absolute ceiling per step (bitcode.yaml: max_step_timeout). 0 = unlimited.
    MaxProcessTimeout  time.Duration // absolute ceiling per process (bitcode.yaml: max_process_timeout). 0 = unlimited.
}

// PoolWithTimeoutConfig combines pool config with timeout config.
// This is what BitCode passes when configuring the full runtime set.
type PoolWithTimeoutConfig struct {
    Pool    PoolConfig
    Timeout TimeoutConfig
}

// Process represents a managed child process.
type Process interface {
    Execute(ctx context.Context, request []byte) ([]byte, error)
    Kill() error
    Pid() int
    Executions() int
    MemoryMB() int
}

// ProcessFactory creates new processes.
type ProcessFactory interface {
    Spawn(ctx context.Context) (Process, error)
}

// Pool manages a set of reusable processes.
type Pool struct {
    config  PoolConfig
    factory ProcessFactory
    procs   chan Process
    mu      sync.Mutex
    closed  bool
    stats   PoolStats
}

// PoolStats tracks pool metrics.
type PoolStats struct {
    TotalExecutions int64
    TotalSpawns     int64
    TotalRecycles   int64
    TotalCrashes    int64
    ActiveProcesses int
    IdleProcesses   int
}

func NewPool(config PoolConfig, factory ProcessFactory) *Pool { ... }
func (p *Pool) Execute(ctx context.Context, request []byte) ([]byte, error) { ... }
func (p *Pool) Stats() PoolStats { ... }
func (p *Pool) Close() error { ... }

// DualPool manages worker + background pools.
type DualPool struct {
    worker     *Pool
    background *Pool
}

func NewDualPool(config DualPoolConfig, factory ProcessFactory) *DualPool { ... }

func (dp *DualPool) Execute(ctx context.Context, poolName string, request []byte) ([]byte, error) {
    switch poolName {
    case "background":
        return dp.background.Execute(ctx, request)
    default:
        return dp.worker.Execute(ctx, request)
    }
}

func (dp *DualPool) Close() error {
    err1 := dp.worker.Close()
    err2 := dp.background.Close()
    if err1 != nil {
        return err1
    }
    return err2
}
```

### 7.2 Pool Lifecycle

```
┌─────────────────────────────────────────────────────────────┐
│ Pool Lifecycle                                               │
│                                                              │
│  1. Pool.Execute(ctx, request)                               │
│     ├── Try get idle process from channel                    │
│     ├── If none available + under capacity → spawn new       │
│     ├── If at capacity → wait (with ctx timeout)             │
│     └── Got process → execute request                        │
│                                                              │
│  2. After execution:                                         │
│     ├── Check MaxExecutions → recycle if exceeded            │
│     ├── Check MaxMemoryMB → recycle if exceeded              │
│     ├── Otherwise → return to idle channel                   │
│     └── If process crashed → spawn replacement (if recovery) │
│                                                              │
│  3. Pool.Close()                                             │
│     ├── Mark closed (reject new requests)                    │
│     ├── Kill all idle processes                              │
│     └── Wait for active processes to finish                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 7.3 Pool is Optional

```go

rt, _ := runtimes.New(
    runtimes.WithPython(),  // no pool config → spawn per call
)

// With pool (production — BitCode):
// Maps 1:1 to bitcode.yaml runtime.worker / runtime.background sections.

    runtimes.WithPython(runtimes.PythonConfig{
        Pool: &runtimes.DualPoolRef{
            Worker: runtimes.PoolWithTimeoutConfig{
                Pool: pool.PoolConfig{
                    Size:            4,
                    MaxExecutions:   1000,
                    MaxMemoryMB:     0,    // 0 = unlimited (overridable by module/process)
                    HardMaxMemoryMB: 0,    // 0 = no ceiling (NOT overridable)
                    CrashRecovery:   true,
                    MaxBackoff:      30 * time.Second,
                },
                Timeout: runtimes.TimeoutConfig{
                    DefaultStepTimeout: 30 * time.Second,  // bitcode.yaml: worker.default_step_timeout
                    MaxStepTimeout:     5 * time.Minute,   // bitcode.yaml: worker.max_step_timeout
                    MaxProcessTimeout:  10 * time.Minute,  // bitcode.yaml: worker.max_process_timeout
                },
            },
            Background: runtimes.PoolWithTimeoutConfig{
                Pool: pool.PoolConfig{
                    Size:            2,
                    MaxExecutions:   100,
                    MaxMemoryMB:     0,
                    HardMaxMemoryMB: 0,
                    CrashRecovery:   true,
                    MaxBackoff:      30 * time.Second,
                },
                Timeout: runtimes.TimeoutConfig{
                    DefaultStepTimeout: 5 * time.Minute,   // bitcode.yaml: background.default_step_timeout
```

**Timeout enforcement flow:**
```
bitcode.yaml                    BitCode Executor              go-json-runtimes
─────────────                   ────────────────              ────────────────
worker:                         1. Read config
  default_step_timeout: 30s     2. Resolve hierarchy          
  max_step_timeout: 5m             (step→process→module→project)
  max_process_timeout: 10m      3. ctx = context.WithTimeout(  
                                     parentCtx, resolvedTimeout)
                                4. runtime.Execute(ctx, ...)  → 4. Respects ctx.Done()
                                                                  Pool enforces:
                                                                  - MaxMemoryMB (recycle)
                                                                  - HardMaxMemoryMB (kill)
```

### 7b.1 bitcode.yaml Runtime Config — UNCHANGED

### 7b.2 Resolution Hierarchy — PRESERVED

BitCode's resolution hierarchy (step → process → module → project → hardcoded) is enforced by **BitCode's executor**, NOT by go-json-runtimes. After extraction:

| Layer | Who Resolves | Where |
|-------|-------------|-------|
| Step-level `timeout` | BitCode executor | `engine/internal/runtime/executor/` |
| Process-level `timeout`, `default_step_timeout` | BitCode executor | Same |
| Module-level `default_step_timeout`, `default_pool` | BitCode module loader | `engine/internal/infrastructure/module/` |
| Project-level (bitcode.yaml) | BitCode config parser | Viper config |
| Hard limits (`max_step_timeout`, `hard_max_memory_mb`) | BitCode executor | Enforced as ceiling |

go-json-runtimes receives:
- `context.Context` with deadline already set (timeout resolved by BitCode)
- `PoolConfig` with limits already set (from bitcode.yaml)
- Per-call `MaxMemoryMB` if overridden by process/module

go-json-runtimes does NOT know about the hierarchy — it just respects what it's given.

### 7b.4 Per-Call Memory Override

When a process or module specifies `max_memory_mb`, BitCode passes it to go-json-runtimes at execution time:

```go
// BitCode executor — per-call override:
execOpts := runtimes.ExecOptions{
    MaxMemoryMB: resolvedMaxMemory,  // from process/module config
    Pool:        resolvedPool,        // "worker" or "background"

result, err := runtimeSet.Execute(ctx, script, function, params, bridge, execOpts)
```

go-json-runtimes enforces: `min(execOpts.MaxMemoryMB, poolConfig.HardMaxMemoryMB)` — the hard limit is never exceeded.

```go
package embedded

import (
    "context"
    "time"

    runtimes "github.com/bitcode-framework/go-json-runtimes"
    "github.com/bitcode-framework/bitcode/internal/runtime/bridge"

// Adapter wraps go-json-runtimes VM to accept *bridge.Context.
// This is the ONLY place where bridge.Context → map[string]any conversion happens.
type VMAdapter struct {
    vm runtimes.VM
}

func (a *VMAdapter) InjectBridge(bc *bridge.Context) error {
    // Convert *bridge.Context to map[string]any using existing BuildGoJSONExtension
    bridgeMap := BuildBridgeMap(bc)
    return a.vm.InjectBridge(bridgeMap)
}

func (a *VMAdapter) InjectParams(params map[string]any) error {
    return a.vm.InjectParams(params)
}

func (a *VMAdapter) Execute(code string, filename string) (any, error) {
    return a.vm.Execute(code, filename)
}

func (a *VMAdapter) Interrupt(reason string) {
    a.vm.Interrupt(reason)
}

func (a *VMAdapter) Close() {
    a.vm.Close()
}

// BuildBridgeMap converts *bridge.Context to map[string]any.
// This is the bridge between BitCode's typed world and go-json-runtimes' generic world.
func BuildBridgeMap(bc *bridge.Context) map[string]any {
    ext := bridge.BuildGoJSONExtension(bc)
    return ext.Functions
}
```

## 9. Security Considerations

### 9.1 Script Sandbox

- Scripts execute in their runtime's sandbox (goja VM, Python subprocess, etc.)
- Bridge functions are the ONLY way scripts interact with the host
- No direct filesystem/network access unless bridge provides it
- Resource limits (timeout, memory) enforced by the runtime

### 9.2 Bridge Security

- Bridge is `map[string]any` — scripts cannot access Go internals
- BitCode builds the bridge with security rules already applied (record rules, env whitelist, etc.)
- go-json-runtimes doesn't know about security — it just passes the map through

### 9.3 Script Path Validation

```go
// Prevent path traversal in script: imports
func validateScriptPath(basePath, scriptPath string) error {
    abs := filepath.Join(basePath, scriptPath)
    abs = filepath.Clean(abs)
    if !strings.HasPrefix(abs, filepath.Clean(basePath)) {
        return fmt.Errorf("script path escapes base directory: %s", scriptPath)
    }
    return nil
}
```

### 9.4 External Runtime Validation

```go
// Node.js version check
func (n *NodeRuntime) Validate() error {
    version, err := getNodeVersion(n.config.Command)
    if err != nil {
        return fmt.Errorf("Node.js not found: %w", err)
    }
    if !meetsMinVersion(version, n.config.MinVersion) {
        return fmt.Errorf("Node.js %s required, found %s", n.config.MinVersion, version)
```

## 8. Security
- Scripts run in runtime sandbox
- Bridge is the only host access mechanism
- Path traversal prevention on script: imports
- Resource limits (timeout, memory) enforced per-runtime
- External runtimes (Node.js, Python) run as child processes

## 9. When to Use What

| Task | Best Runtime | Why |
|------|-------------|-----|
| Quick validation, transforms | goja (.js) | Fastest startup, embedded, no overhead |
| Modern JS (async/await, modules) | quickjs (.js) | Full ES2023, still embedded |
| Concurrent processing, batch ops | yaegi (.go) | Goroutines, type safety |
| npm packages (Puppeteer, Excel) | Node.js (.ts) | Full npm ecosystem |
| ML, data science, pandas | Python (.py) | Full pip ecosystem |
| High-performance, portable | WASM (.wasm) | Sandboxed, any source language (Phase 4.5k) |
| Maximum native performance | Native (.so) | Direct Go function calls (Phase 4.5k) |

## 10. Troubleshooting

### Common Errors
- "no script runtime registered for extension '.py'" → Python not installed or not in PATH
- "Node.js 20.0 required, found 18.x" → Upgrade Node.js
- "script path escapes base directory" → Path traversal attempt blocked
- "no script runtime registered for extension '.wasm'" → WASM runtime not enabled (Phase 4.5k)
